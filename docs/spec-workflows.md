# מפרט תהליכים (workflows) — גרסה נבנית 2026-06-25

כל הדרישות שהמנכ"ל הגדיר. זהו מקור האמת לפיצ'רים האלה.

## תפקידים (תזכורת)
מנהל (admin) · אחראי כלים (manager) · בעל עגלה (cart_owner, צפייה בלבד בעגלה שלו).

## 1. חתימה יומית על עגלה
- בעל העגלה מבצע **בדיקה אחת ביום** וחותם על העגלה שלו.
- כל יום בלי חתימה = יום שבו הוא **לא נוכח** (אינדיקציה לנוכחות).
- נשמר רישום `signoffs` לכל עגלה/יום/משתמש.

## 2. בדיקת אחראי/מנהל אחת לרבעון ("עגלות בקו")
- אחת לרבעון אחראי או מנהל בודק את העגלות שבקו וחותם. (עודכן מחודשיים לרבעון לבקשת המנכ"ל, 2026-06-25.)
- אם לא בוצע בתוך **90 יום** → **כל העגלה אדומה**.
- נשמר רישום `inspections`.

## 3. כיול — בקשה ואישור → "חוסר מאושר"
- כלי מכויל שנכנס ל-**30 הימים האחרונים** לתוקפו → נפתחת לבעל העגלה אופציה "בקש שליחה לכיול".
- בחלון הבקשה הוא רואה את הכלים המכויילים שלו עם התקפים, ויכול לבחור **את כולם** או **רק את מה שפג/קרוב**.
- סימון → נשלחת **בקשה** (status=pending). מאשר = מנהל / אחראי / משתמש אחר (לא המבקש).
- לאחר אישור → **חוסר מאושר** (approved) → אפשר להוציא לכיול (loc=כיול).
- **קיצור:** אם מנהל/אחראי מחליט להוציא לכיול בעצמו → **אין צורך בשני אימותים**, ישר חוסר מאושר.

## 4. חוסר מאושר לשימוש מחוץ למחלקה
- בעל עגלה יכול לבקש חוסר מאושר על כל כלי בעגלה, **עם סיבה = המיקום** שבו ישתמש בו מחוץ למחלקה.
- אותו מנגנון אישור (בקשה → אישור → חוסר מאושר, loc = המיקום/חוסר).

## 5. הצהרת כלי כשבור
- אופציה להצהיר כלי כ**שבור** → loc=שבור.
- נשלחת **התראה למנהלים**, והכלי נכנס ל**דוח שבורים עם מק"ט**.
- אחר כך: או חוסר מאושר עד שיגיע מחליף, או השלמה מגיבוי במלאי.

## 6. כלי במיקום לא ידוע
- אפשר להכניס כלי **בלי לדעת את מיקומו**.
- אם הוא נשאר במצב "לא ידוע" **מעל שנה** → התראה שדורשת מהמנהל **לבטל אותו**.

## רעיונות שדרוג שנוספו תוך כדי (להמשך)
- היסטוריית תנועות לכל כלי (כיול→חזרה→חוסר) כ-timeline.
- לוח התראות מרוכז למנהל (כיול, שבורים, עגלות אדומות, נוכחות).
- דוח נוכחות חודשי לפי חתימות יומיות.
- סריקת QR/ברקוד לכלי מהטלפון לאיתור/אישור מהיר.
- "מצב גיבוי" — כלי גיבוי במלאי לכל מק"ט נפוץ, להשלמה אוטומטית בשבור.
